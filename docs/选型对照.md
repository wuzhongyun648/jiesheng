# 选型对照：markdown 文件 vs Mem0 vs MemoryOS

用途：说明结绳为什么用 markdown 文件、而不上 Mem0 / MemoryOS 这类向量或图记忆引擎，并给出规模上来后的切换路径。

判断口径是结绳当前的场景：科研记忆层，几十到上百篇论文，强溯源，本地优先，跑在 QClaw 技能里，受安全扫描约束，要求纯本地、无 key、无网络。

---

## 一、三个选项分别是什么

markdown 文件，结绳现用，对应 LLM Wiki 式做法。一篇论文一个 `.md`，字段化存储出处、核心假设、关键结论、我的判断、关联；用 `[原文 §x]` 和 `[推断]` 做字段级溯源；关系用有向边显式建模，分冲突、互补、取代、可组合。检索靠扫描文件，几十到上百篇直接加载或关键词即可。零依赖、纯本地、人可读可改，是一个可累积、可审计的知识库。

Mem0，arXiv:2504.19413，ECAI 2025，Apache-2.0。面向生产的对话记忆层，从对话里抽取、整合、检索关键信息，存进向量库；图变体 Mem0ᵍ 用图表示刻画对话元素之间的关系。论文报告在 LoCoMo 上相对 OpenAI 记忆约高 26%，p95 延迟比全上下文低 91%，token 成本省九成以上。它的定位是让 Agent 跨会话记住用户说过的事和偏好。

MemoryOS，arXiv:2506.06326，EMNLP 2025 Oral，北邮 BAI-LAB。借鉴操作系统内存管理的分层记忆系统：存储、更新、检索、生成四个模块，加短期、中期、长期三级存储；短到中按对话链 FIFO 更新，中到长用分段分页；长期记忆里维护用户知识库和 Agent 画像，检索按热度打分，热度看交互次数、访问次数、时近性。支持 ChromaDB、Docker、MCP、BGE-M3、Qwen3 embedding，PyPI 可装。定位是个性化对话记忆加用户画像。

一个关键区分：结绳的 markdown 记的是这些论文，是知识；Mem0 和 MemoryOS 记的是这个用户，是对话。后两者本就是对话记忆，抽的是用户偏好和事实，不是带溯源的论文知识，也没有你策划的冲突关系。

## 二、对照表

| 维度 | markdown 文件，现用 | Mem0 | MemoryOS |
|---|---|---|---|
| 本质定位 | 可累积、可审计的知识库 | 生产级对话记忆层，向量加可选图 | OS 式分层对话记忆加用户画像 |
| 存储形态 | 纯文本 .md 文件 | 向量库，Mem0ᵍ 加图 | 向量库加分级分页结构 |
| 可读、可审计 | 是，人可直接读和改 | 否，嵌入或 DB | 否，嵌入或 DB |
| 溯源，原文 vs 推断 | 字段级、强制、自己掌控 | 抽事实，无原文与推断之分，无章节级引用 | 同 Mem0 |
| 关系与冲突检测 | 显式有向边，Agent 判定并双向回写 | Mem0ᵍ 有图，但来自对话自动抽取，非论文冲突 | 无冲突边概念 |
| 检索与规模拐点 | 几十到上百篇加载或关键词即可，更多需换后端 | 语义向量检索，规模、高并发、低延迟下才显价值 | 语义加热度排序，长对话、海量下才显价值 |
| 依赖与部署 | 零依赖、纯本地、无 key、无网络，过 QClaw 扫描 | 需 embedding、向量库、抽取 LLM，通常带 key 或网络 | 同 Mem0，需 Chroma、BGE、Docker |
| 用户画像 | 非重点，结绳记论文不记用户 | 有用户记忆 | 有显式用户与 Agent 画像，但对结绳价值低 |

## 三、结论：当前规模选 markdown，且是主动选择

在结绳现在的口径下，markdown 在几条最关键的维度上占优：

- 可审计加溯源是结绳的信任基础，而这正是 Mem0 / MemoryOS 的弱项。它们抽的是不可读的事实，没有 `[原文]` 与 `[推断]` 这种字段级、可回原文核对的机制。把记忆埋进向量库，就失去了记忆可见、可逐条核对这一点。
- 关系与冲突。结绳要的是论文级冲突边，比如平稳 vs 非平稳，由 Agent 判定、双向回写。Mem0ᵍ 的图来自对话自动抽取的实体关系，对象和粒度都不同；MemoryOS 没有冲突边的概念。
- 依赖与部署。结绳跑在 QClaw 技能里，纯本地、无 key、无网络既是约束也是优点。引入 Mem0 / MemoryOS 要么带 key 和网络，要么为纯本地自搭模型加向量库，都会破坏这一点。
- 规模没到拐点。向量检索、热度排序、分层分页要到海量条目、高并发、低延迟才划算。几十到上百篇，加载或关键词就够，过早引入重栈是负担。

Mem0 / MemoryOS 真正强的地方是语义检索规模化、用户画像、自动关系图，这些对结绳要么不是核心，要么要等规模上来才用得上。所以现在不上是判断，不是能力不足。

---

## 四、切换路径，schema 不变

结绳预留了平滑升级。markdown 始终是真相源；需要时把 Mem0 / MemoryOS 接成它背后的检索或索引层，从同一批字段喂数据，论文 schema 不动。

当论文多到几百上千、或需要低延迟语义检索时，加一个 `build_index` 步骤：读 `memory/papers/*.md`，把核心假设、关键结论、标签灌进 Mem0 / MemoryOS 做嵌入和检索，把关联边喂给 Mem0ᵍ 的图；查询走引擎，写入仍先落 markdown。被替换的只是 `list_papers.py` 背后的检索后端，字段、文件、溯源标签都不动。

---

注：LoCoMo 这类基准的分数会随版本和评测设置大幅波动，同一基准上不同论文给 Mem0 的总分从六十多到九十以上都有。所以选型按机制和场景是否匹配来判断，比看谁分高更稳。
